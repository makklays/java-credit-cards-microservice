
Методы проверки поведения (Mockito Verify)

- verify(mock)  — проверяет, был ли вызван конкретный метод у мока ровно один раз.
- times(n)      — задает точное количество раз, которое должен был выполниться метод.
- never()       — гарантирует, что проверяемый метод вообще не вызывался в ходе теста.
- atLeastOnce() — проверяет, что метод был вызван как минимум один раз или чаще.
- atLeast(n) / atMost(n) — задает минимальные или максимальные границы для количества вызовов метода.
- inOrder()     — проверяет, что методы у разных моков вызывались в строго определенной последовательности.
- verifyNoInteractions(mock) — подтверждает, что к указанному моку вообще не было никаких обращений.
- verifyNoMoreInteractions(mock) — проверяет, что после выполненных проверок у мока больше не вызывалось никаких других методов.

Методы настройки заглушек (Mockito Stubbing)

- when() — запускает цепочку настройки мока, определяя, как он должен себя вести при вызове.
- thenReturn() — задает конкретное значение, которое мок должен вернуть в ответ на вызов.
- thenThrow() — заставляет мок выбросить указанное исключение (ошибку) при вызове его метода.
- doReturn() / doThrow() — альтернативный синтаксис настройки, необходимый для работы с void-методами и шпионами (spies).
- any() / anyString() / anyLong() — выступают аргументами-заглушками, позволяя моку срабатывать на любые входящие данные соответствующего типа.

Методы проверки результатов (JUnit 5 Assertions)

- assertEquals() — проверяет, совпадает ли ожидаемый результат с фактическим значением.
- assertNotEquals() — убеждается, что полученное значение отличается от нежелательного результата.
- assertTrue() / assertFalse() — проверяет, является ли переданное логическое условие истинным или ложным.
- assertNull() / assertNotNull() — контролирует наличие или отсутствие ссылки на объект (null).
- assertSame() — проверяет, указывают ли две переменные на один и тот же физический объект в памяти.
- assertThrows() — проверяет, что тестируемый код падает с конкретным типом исключения при определенных условиях.
- assertAll() — группирует несколько проверок вместе, позволяя увидеть результаты каждой из них, даже если одна из первых упала.

Правильная комбинация when() и verify() строится на классической структуре теста AAA (Arrange, Act, Assert) или GWT (Given, When, Then).
Главное правило: when() всегда пишется ДО вызова тестируемого метода, а verify() — всегда ПОСЛЕ.

Пошаговая схема комбинации
1. Arrange (Подготовка):
   Вы настраиваете мок-зависимость. Вы говорите: «Если в будущем процессе теста у этого мока вызовут метод А, то верни значение Б».
2. Act (Действие):
   Вы запускаете реальный метод вашего тестируемого класса. Внутри себя он вызовет настроенный ранее мок.
3. Assert (Проверка):
   Вы проверяете факт взаимодействия. Вы спрашиваете: «А действительно ли мой класс в процессе работы вызвал метод А у мока?
   И сделал ли он это правильное количество раз?».

@Test
void shouldProcessPaymentSuccessfully() {
    // 1. ПОДГОТОВКА (when)
    // Заранее говорим моку вернуть true, если его спросят про заказ №10
    when(paymentClient.charge(10L, 500.0)).thenReturn(true);

    // 2. ДЕЙСТВИЕ
    // Вызываем реальный метод сервиса. Внутри него лежит логика,
    // которая должна обратиться к paymentClient.charge()
    orderService.completeOrder(10L, 500.0);

    // 3. ПРОВЕРКА (verify)
    // Проверяем, что сервис не просто вернул результат,
    // но и реально сходил в платежную систему ровно 1 раз
    verify(paymentClient, times(1)).charge(10L, 500.0);
}
